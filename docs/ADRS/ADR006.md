ADR 006: Estructura de Directorios Híbrida (Feature-Sliced Frontend, Layered Backend)
Fecha: 18 de marzo de 2026

Estado: Aceptado

Contexto: La organización del código fuente impacta directamente en la velocidad a la que el equipo puede encontrar archivos y añadir nuevas funcionalidades. Una arquitectura sobre-diseñada (como Domain-Driven Design estricto) generaría demasiados archivos vacíos para nuestro MVP. Sin embargo, un enfoque totalmente plano terminaría en un código espagueti.

Decisión: Adoptar una estructura híbrida:

Frontend (React/Vite): Utilizaremos una arquitectura orientada a funcionalidades (Feature-Sliced Design simplificado), agrupando componentes, hooks y utilidades específicos de un dominio (ej. features/arbitrage) en un solo lugar, separados de la UI genérica.

Backend (Node.js): Utilizaremos una arquitectura clásica de capas (Layered Architecture por responsabilidades técnicas: db, services, sockets), dado que el backend del MVP funciona como una tubería (pipeline) de datos lineal sin lógica de dominio ramificada.

Consecuencias:

Positivas: * El Frontend se mantiene escalable; si en la Fase 2 se añade una "Calculadora de Cobertura", se encapsulará en su propia carpeta features/hedging sin romper el código de arbitraje.

El Backend se mantiene simple, plano y fácil de entender. La separación clara entre los servicios de datos (services/marketData) y los WebSockets (sockets/spreadHandler) facilita la depuración de la ingesta en tiempo real.

Negativas: Requiere una breve alineación del equipo para entender que el patrón de organización cambia dependiendo de si están trabajando en el cliente o en el servidor.
