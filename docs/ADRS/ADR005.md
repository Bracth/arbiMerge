ADR 005: Adopción de Prisma como ORM para PostgreSQL
Fecha: 18 de marzo de 2026

Estado: Aceptado

Contexto: En el entorno de alta presión de una hackathon, interactuar con la base de datos mediante consultas SQL crudas (raw SQL) es propenso a errores de tipado, fallos de sintaxis y ralentiza la iteración si el esquema de datos necesita cambiar rápidamente. Necesitamos una forma rápida, segura y tipada de conectar nuestro backend Node.js con PostgreSQL.

Decisión: Utilizar Prisma ORM como capa de abstracción para la base de datos.

Consecuencias:

Positivas: * Iteración ultrarrápida del esquema de base de datos mediante el archivo schema.prisma.

Generación automática de un cliente de base de datos con autocompletado y seguridad de tipos (Type Safety), reduciendo drásticamente el tiempo de depuración.

Eliminación de código repetitivo (boilerplate) para operaciones CRUD básicas.

Negativas / Riesgos: * El motor de consultas de Prisma (Query Engine) depende de binarios específicos del sistema operativo. Esto introduce un riesgo de fallo al empaquetar la aplicación en Docker (ej. compilar en Mac/Windows y desplegar en Linux).

Mitigación: Deberemos configurar explícitamente el binaryTargets en el schema.prisma y asegurar que la generación del cliente (prisma generate) se ejecute dentro del proceso de construcción (build) del Dockerfile, no en la máquina anfitriona.
